  - Goals are stored as goal objects within the system database with attributes including goal name, category (performance/development), description, target metrics, deadlines, progress status, and visibility.
  - Activities and achievements are stored as discrete entities or records that can be linked to one or more goal objects.
  - Continuous Performance architecture uses relational entities where many activities can link to single or multiple goals, a many-to-many relationship managed through join tables or link objects.
- **Page Architecture:**
  - The Continuous Performance page is built on SAP SuccessFactors’ UI framework, leveraging UI5 libraries (SAP Fiori design principles) for responsive, dynamic user interface elements.

## 2. Structure and Relationship of Goal Plans and Individual Goals

SAP SuccessFactors uses a hierarchical and modular data model for goal management:
- **Goal Plan:** A container object or schema grouping multiple individual goals. It represents a structured framework for managing related goals collectively.
- **Default Goal Plan:** Every user has a default goal plan, typically comprising essential performance and development goals.
- **Multiple Goal Plans:** Enterprises can define multiple plans for different contexts (e.g., annual reviews, project goals, skill development).
- **Relationship:**
  - Each goal belongs exclusively to one goal plan.
  - Through the UI framework, users can view goals by plans; goal plans render as collections or panels on the interface using SAP UI5 controls for grouping and tab navigation.
- **Data Model:**
  - Goals and goal plans are linked using foreign key references.
  - The framework supports permissions and visibility rules at the goal plan and goal level.
- **UI Framework:**

- One single development goal within the same plan.
This restriction was enforced through UI and backend logic checks:
- **Backend:**
  - Database schema allowed only a single foreign key linking an activity record to one goal at a time.
  - API endpoints validated inputs to restrict multiple goal associations.
- **Frontend:**

  - UX workflows prevented multiple selections, explicitly confirming single linkage.
- **Implications:**
  - Limited flexibility in tracking cross-functional contributions.
